# AlphaFlow US 트레이딩 시스템 기술 평가 및 실전 로드맵

> 기준 코드베이스: `/api`, `/frontend`, `docker-compose.yml`, `api/models/schema.sql`, `scripts/init_db.py`, `README.md`  
> 평가 원칙: 칭찬 최소화, 리스크 중심, 실전 운용 가능성 기준

---

## Part 1: 냉정한 기술 평가

### 1) 아키텍처 설계 — **등급: C-**

현재 구조는 기능 분리는 되어 있으나, 운영 관점에서는 "모듈화된 단일 프로세스" 수준입니다. `api/services/batch.py`의 `run_full_batch()`가 Step 0~4를 직렬 실행하고, `api/services/scheduler.py`는 해당 배치와 자동매매/뉴스수집/매크로 작업을 병렬 스케줄링합니다. 문제는 작업 간 자원 충돌과 실패 격리가 충분히 설계되지 않았다는 점입니다.

핵심 리스크는 세 가지입니다.

- **의존성의 암묵적 결합**: Step 1 스크리닝 결과가 Step 3 분석 대상을 사실상 결정하고(`batch.py`), Step 2 뉴스/인덱싱 상태가 Step 3 텍스트 분석 품질을 좌우합니다. 하지만 실패 시 보상 트랜잭션, 재시도 큐, dead-letter 같은 장애 격리 계층이 없습니다.
- **스케줄러 충돌 가능성**: `scheduler.py`에서 5분/10분/30분/1시간/장중 배치가 동시에 작동합니다. `run_full_batch()`와 `auto_trade_loop()`가 모두 signals/trades를 다루는 동안 동시성 제어가 없습니다. DB 수준 락 정책이나 작업 중복 방지 토큰도 없어 레이스 컨디션 위험이 높습니다.
- **에러 전파의 비일관성**: 많은 함수가 `try/except`로 에러를 로그 후 무시합니다. 이는 프로세스 크래시를 줄이지만, 반대로 "실패를 성공처럼 보이게" 만들어 운영자가 장애를 놓치게 합니다.

요약하면, 서비스 분리는 되어 있으나 **작업 오케스트레이션 계층이 약한 C- 수준**입니다. 현재 상태에서는 "동작하는 파이프라인"이지 "장애 허용 파이프라인"은 아닙니다.

### 2) 데이터 파이프라인 품질 — **등급: D+**

#### 뉴스 수집 품질
`api/services/news_crawler.py`는 Finnhub + Yahoo RSS + Google RSS를 결합합니다. 소스 다양성 자체는 장점처럼 보이지만, 실제로는 소스 품질 편차가 큰데 가중치/신뢰도 계층이 없습니다.

- Finnhub는 상대적으로 구조화된 데이터이나 플랜 제한/레이트 리밋 영향이 큼.
- Yahoo/Google RSS는 노이즈와 중복이 많고, 종목 연관성(특히 ticker ambiguity) 오류가 빈번할 수 있음.
- `_url_hash(url)` 기반 중복 제거는 URL 파라미터 변형/리다이렉트 URL에 취약.

#### 가격 데이터 품질
`api/services/price_crawler.py`는 `yfinance.download(..., period="1y")`로 50개 배치 조회합니다. 이 방식은 구현은 단순하지만 실전 트레이딩 데이터 공급원으로는 취약합니다.

- yfinance는 비공식 소스 특성상 지연/결측/일시 오류가 빈번.
- 배치 실패 시 배치 단위로 오류 처리(`continue`)되어 갱신 공백이 누적될 수 있음.
- 데이터 스테일(stale) 검증이 없어 오래된 가격이 의사결정에 섞일 가능성.

#### 정합성/결측/중복 처리
- `news_articles.url_hash` 유니크는 있으나 기사 의미 중복(다른 URL, 같은 본문) 방지 미흡.
- `signals`는 유사 시그널 중복 삽입 가능 구조(유니크 제약 없음).
- 결측 발생 시 neutral fallback이 많아, 데이터 품질 저하가 모델 중립화로 은폐됩니다.

즉, "수집은 되지만 품질 보증이 없는" 상태입니다. 운영 지표를 만들기 위한 최소선은 넘었으나 실전 매매 입력 데이터로는 D+ 평가가 타당합니다.

### 3) AI/ML 모델 활용 — **등급: D**

#### 키워드 감성 분석
`api/services/sentiment.py` 기반 1차 감성은 규칙 기반으로 빠르지만, 금융 문맥(부정어, 역설, 선반영 표현)을 다루기 어렵습니다. 예: "실적 부진 우려 완화" 같은 문장은 키워드 합산으로 오판 가능성이 큽니다.

#### Ollama 로컬 LLM(qwen3)
`ollama_client.py` + `rag_analyzer.py` 조합은 구조적으로는 잘 묶였지만, 모델 금융 특화 검증이 없습니다. 또한 JSON 강제 출력 실패를 `_parse_json_response()`에서 다단 파싱으로 복구하는 구조는, 사실상 모델 출력 신뢰도가 낮다는 반증입니다.

#### RAG 효용
`news_indexer.py`의 `_chunk_text()`는 긴 텍스트를 사실상 첫 500자만 사용합니다. 이는 기사 후반 핵심 문맥을 버리는 설계이며, 임베딩 검색 품질을 직접적으로 저하시킵니다. 검색 결과를 LLM에 넘기는 부분도 리랭킹/신뢰도 컷오프가 없습니다.

#### 비전 모델 차트 분석
`chart_analyzer.py`는 `mplfinance` 이미지 + 비전 LLM 패턴 인식을 사용합니다. 하지만 전통적 패턴 검출기(규칙/통계)와 교차검증이 없고, 결과의 캘리브레이션/백테스트 근거가 없습니다. 실전 신호 축으로 쓰기에는 신뢰도가 낮습니다.

#### Gemini Embedding 이중 컬렉션 문제
문제 설명상 Gemini 임베딩이 추가되었지만 현재 코드베이스에는 `gemini_client.py`, `gemini_indexer.py`, `geopolitical_engine.py`, `fulltext_crawler.py`가 확인되지 않습니다. 즉 **설계 문서와 구현이 불일치**합니다. 이 상태에서 bge-m3/Gemini 이중 컬렉션을 운영하면 지표 해석/디버깅 혼선이 커집니다.

결론: AI 요소가 많지만 검증 체계가 약해 "모델 중심 자동매매"를 뒷받침하지 못합니다. D 등급입니다.

### 4) 트레이딩 로직(핵심) — **등급: D+**

#### 스크리닝 기준(score ≥60 or ≤35)
`batch.py`의 기준은 하드코딩이며 통계적 근거가 코드/문서에 없습니다. 중간 점수 구간을 폭넓게 제외하는데, 이는 거래 기회 손실 혹은 데이터 수집 편향을 만들 수 있습니다.

#### 3축 가중 합산(text 35 / numeric 50 / macro 15)
`trading_engine.py`에서 settings 기반으로 읽지만, 가중치 합 1.0 검증이 없습니다. 설정 오류 시 의사결정 스케일이 무너질 수 있습니다.

#### BUY/SELL 임계값(70/30)
임계값이 고정이며 변동성/시장 레짐/종목군에 따른 적응이 없습니다. 결국 시장 상태가 바뀌면 시그널 분포가 붕괴할 수 있습니다.

#### 주문 안전장치
설정값(`max_order_amount`, `daily_order_limit`, `max_exposure_pct`)은 존재하나, 이것만으로는 리스크 관리가 부족합니다.

- 종목/섹터 집중도 제한 부재
- 상관관계 기반 익스포저 제어 부재
- 장중 유동성/슬리피지 방어 로직 미흡

#### 손절/익절(-8/+15) 및 레버리지 ETF
상수 규칙은 단순명료하지만 변동성 정규화(ATR 기반), 이벤트 리스크 회피, 장 마감/갭 리스크 처리 등이 없습니다. 레버리지 ETF(TQQQ/SQQQ)는 레짐 오판 시 손실이 급격히 확대될 수 있는데, 현재 매크로 입력 자체가 조잡해 리스크가 큽니다.

핵심적으로 **신호 생성보다 포지션 리스크 제어가 약한 구조**입니다.

### 5) 시그널 검증 체계 — **등급: C-**

`performance_tracker.py`는 signal_performance, daily_snapshot, weekly_report를 생성하며 "운영 후 성과 측정"의 틀은 있습니다. 하지만 정교한 검증 체계로 보기에는 한계가 명확합니다.

- `update_signal_performance()`는 yfinance 기반 사후 수익률 계산으로 체결 슬리피지/수수료/시장충격 반영이 없음.
- hit 정의가 방향성 중심이라 위험조정 수익을 설명하지 못함.
- `generate_weekly_report()`의 샤프 계산은 표본이 매우 작고(주간 구간), 정규성/독립성 가정을 만족하기 어려움.
- 백테스트 엔진이 없어 전략 변경 시 재현 가능한 비교 실험이 불가능.

즉, 모니터링 지표는 있으나 "전략 타당성 검증 체계"는 아닙니다.

---

### 6) 국제정세 엔진(신규) — **등급: F (현 코드 기준)**

요청 설명에는 geopolitical 엔진/테이블/스케줄이 제시되어 있으나, 현재 레포지토리에서는 관련 서비스 파일과 스키마(`geopolitical_events`, `geopolitical_regime`)를 확인할 수 없습니다. 구현이 누락되었거나 반영 전 상태입니다.

이 항목은 기능 품질을 논하기 전에 **구현 존재성 자체가 불충분**합니다.

- RSS 기반 지정학 분석의 노이즈 필터링, 카테고리 분류 근거, 지연 보정 로직을 코드로 검증할 수 없음.
- macro_engine과 결합된 증거도 없음.

따라서 냉정하게 F가 맞습니다(문서상 존재, 코드상 부재).

### 7) 인프라 & 운영 — **등급: D**

#### Docker Compose
`docker-compose.yml`은 로컬 개발에는 유용하지만 프로덕션 적합성은 낮습니다.

- 앱이 `--reload`로 실행됨(개발 모드).
- DB 비밀번호 기본값이 노출됨.
- 모니터링/백업/롤링 배포/자원 제한 설정 없음.

#### 로깅/모니터링/알림
`main.py` 로깅은 기본 stdout 형식만 존재. 구조화 로그, 알림 연동, 장애 페이지 개념이 없습니다. 스케줄 실패가 누적되어도 운영자가 늦게 인지할 가능성이 큽니다.

#### 보안
- CORS `allow_origins=["*"]` + `allow_credentials=True`는 위험 조합.
- 비밀키는 .env 의존이며 비밀 관리 솔루션(KMS/Vault) 부재.
- 레이트 리미팅/인증 계층이 없어 민감 엔드포인트(`/api/batch/run`) 오남용 가능.

#### 장애 복구
- DB 다운/외부 API 한도 초과/스케줄러 중단 시 자동 복구 시나리오가 미흡.
- 작업 재처리 큐 및 상태 머신이 부족.

현 단계는 "개발자 상주 운영" 전제를 깔아야 유지 가능한 D 수준입니다.

### 8) 코드 품질 — **등급: C**

#### 스타일/타입/핸들링
- 비동기 중심 구조와 로깅 관례는 비교적 일관적.
- 그러나 Optional/타입 엄밀성, fallback 남용, broad exception 처리로 정확성이 떨어집니다.

#### 테스트
레포지토리 내 테스트 인프라가 사실상 없습니다. 실제로 환경에서 `python -m pytest -q` 실행 시 pytest 모듈 자체가 없어 테스트 체계 부재를 확인했습니다.

#### 설정 관리
`.env`와 DB `settings` 이중화는 유연하지만 우선순위 규약/검증이 약하면 설정 드리프트를 유발합니다.

#### 기술 부채
- README와 구현 간 괴리(신규 모듈 미반영)
- 실험적 로직(차트 LLM, 임계값 하드코딩)의 검증 부재
- 운영 자동화 부족

따라서 코드베이스는 "빠르게 구축된 MVP" 관점의 C가 적절합니다.

## Part 2: 종합 점수 & 현실 진단

### 1) 현재 성숙도 판정: **MVP (초기 운용 가능, 실전 자금 운용 불가)**

이 시스템은 PoC를 넘어 "동작하는 제품" 형태(MVP)에는 도달했습니다. 이유는 다음과 같습니다.

- DB 스키마, FastAPI 라우터, 스케줄러, 주문 경로가 연결되어 end-to-end 동작 가능
- 성과 추적 테이블과 리포트 생성 등 운영 지표의 최소 형태 존재
- 설정 테이블을 통한 런타임 파라미터 변경 가능

그러나 Beta/Production 단계로 보기 어려운 결정적 이유도 분명합니다.

- 테스트/검증/백테스트 기반이 사실상 없음
- 데이터 품질과 모델 출력 신뢰성의 계량적 증명 부재
- 운영 보안/복구/알림 체계가 약함
- 문서와 구현 불일치(신규 기능 누락)로 변경 관리 신뢰도 저하

즉, "아이디어를 서비스 형태로 실행한 MVP"이지, 고객 자산을 맡기는 자동매매 플랫폼은 아닙니다.

### 2) 이 시스템으로 실제 돈을 걸어도 되는가?

## **No**

#### 구체적 근거
1. **전략 유효성 미검증**: 백테스트/워크포워드/체결비용 반영 검증이 없어 기대수익의 통계적 근거가 없음.
2. **신호 신뢰성 취약**: 감성+RAG+비전 결과가 파싱 실패/노이즈/편향에 쉽게 흔들림.
3. **리스크 관리 불충분**: 포지션 사이징·상관관계·섹터 집중·시장 급변 보호 체계 부족.
4. **운영 리스크 과대**: 스케줄 충돌, 장애 알림 미비, 복구 시나리오 약함.
5. **보안/통제 미흡**: CORS 설정, 인증·레이트리밋 부재로 운영 중 사고 가능성.

자금을 투입한다면 현재는 **Paper Trading 전용**으로 제한하는 것이 합리적입니다.

---

### 3) 가장 시급히 해결해야 할 Top 5 (우선순위)

1. **백테스트/워크포워드 검증 프레임워크 구축 (P0)**  
   - 현재 임계값/가중치가 임의값이라 전략 근거가 없습니다.
2. **리스크 관리 엔진 고도화 (P0)**  
   - 변동성 기반 포지션 사이징, 총/섹터/상관 익스포저 캡 도입 필수.
3. **데이터 품질 게이트 도입 (P0)**  
   - 스테일 데이터 차단, 기사 중복/노이즈 필터, 소스 신뢰도 점수화.
4. **운영 안전성(알림·복구·동시성 제어) 강화 (P1)**  
   - 스케줄러 작업 잠금, 재시도 큐, 장애 알림 없이는 장기운영 불가.
5. **AI 신호 검증/교정 체계 구축 (P1)**  
   - LLM 출력 스키마 검증, confidence gating, 모델 성능 벤치마크 필요.

## Part 3: 실전 로드맵 (Phase 1 ~ 5)

> 최종 목표: **월간 양의 알파(benchmark 대비) + 최대 낙폭(MDD) 15% 이내 통제**

### 전체 의존성 맵
- Phase 1(관측/통제) 없이는 Phase 2(검증) 신뢰 불가
- Phase 2(백테스트/평가) 없이는 Phase 3(모델 확장) 금지
- Phase 3(신호 개선) 없이 Phase 4(라이브 준비)는 위험 과대
- Phase 4(운영통제) 검증 통과 후에만 Phase 5(점진 라이브)

---

## Phase 1 — 운영 안정화 & 데이터 신뢰성 확보 (예상 4~6주)

### Step 1-1. 스케줄러 충돌 제어
- **목적**: 배치/자동매매/인덱싱 동시 실행 충돌 제거
- **구현 내용**: 작업별 락 키(예: redis/postgres advisory lock), 중복 실행 방지, 장시간 작업 timeout 정책
- **DoD**: 동일 job id 동시 실행 0건, 충돌 로그 0건(2주)
- **난이도**: Mid

### Step 1-2. 데이터 품질 게이트
- **목적**: 잘못된 입력 데이터가 신호로 들어가는 경로 차단
- **구현 내용**: 가격 staleness 체크(예: N분 이내), 뉴스 소스 신뢰도 점수, URL 정규화 + 본문 해시 dedup
- **DoD**: 결측/중복/스테일 데이터 비율 대시보드화, 임계치 초과 시 신호 생성 중단
- **난이도**: Mid

### Step 1-3. 에러 분류/알림 체계
- **목적**: "실패했는데 모르는" 상태 제거
- **구현 내용**: 에러 코드 분류(외부API/DB/모델/파싱), Slack/이메일/PagerDuty 연동, 재시도 정책
- **DoD**: P1/P2 장애 알림 SLA(5분 이내), 주간 장애 리포트 자동 생성
- **난이도**: Mid

### Step 1-4. 보안/접근통제 기초
- **목적**: 운영 사고 및 API 오남용 방지
- **구현 내용**: CORS 화이트리스트, 인증 토큰, 민감 엔드포인트 rate limit, secret rotation 정책
- **DoD**: 외부 호출 인증 필수, 무인증 실행 가능한 관리자 API 제거
- **난이도**: Mid

### Phase 1 체크포인트
- 배치 실패율 < 3%
- 데이터 게이트 차단율/원인 리포트 확보
- 장애 탐지-알림-조치 타임라인 측정 가능

---

## Phase 2 — 백테스트 프레임워크 & 시그널 품질 검증 (예상 6~8주)

### Step 2-1. 이벤트 드리븐 백테스터 구축
- **목적**: 전략 성능을 재현 가능하게 검증
- **구현 내용**: 바(OHLCV) 기반 체결 시뮬레이터, 수수료/슬리피지/체결지연 모델, 포지션/현금 장부
- **DoD**: 동일 입력 재실행 시 동일 결과, 주요 전략 지표 자동 산출
- **난이도**: High

### Step 2-2. 신호 라벨링/평가 파이프라인
- **목적**: 신호가 실제 수익 기여하는지 정량 검증
- **구현 내용**: horizon별 라벨(1d/5d/10d), precision-recall, IC(Information Coefficient), decile 분석
- **DoD**: 신호 품질 리포트 주기 발행, 기준 미달 시 자동 비활성화
- **난이도**: High

### Step 2-3. 파라미터 탐색/과최적화 방지
- **목적**: 60/35, 70/30, 가중치의 근거 확보
- **구현 내용**: walk-forward split, purged k-fold, 파라미터 안정성 heatmap
- **DoD**: out-of-sample 성능이 in-sample 대비 일정 기준 유지
- **난이도**: High

### Step 2-4. 리스크 지표 표준화
- **목적**: 수익률 중심 의사결정에서 위험조정 의사결정으로 전환
- **구현 내용**: Sharpe/Sortino/Calmar, rolling MDD, tail-risk(CVaR), turnover 비용 지표
- **DoD**: 전략 승인 기준표(예: MDD<15%, 월간 alpha>0, turnover 제한) 문서화
- **난이도**: Mid

### Phase 2 체크포인트
- "백테스트 없음" 상태 해소
- 기본 전략의 OOS 성능/리스크 보고서 확보
- 임계값/가중치 변경 시 회귀검증 가능

---

## Phase 3 — 모델 스택 개선 (LSTM 포함) 및 앙상블화 (예상 8~12주)

### Step 3-1. LSTM/시계열 모델 도입
- **목적**: 규칙 기반 신호 보완 및 시계열 패턴 반영
- **구현 내용**: 종목별/클러스터별 시계열 특성(returns, vol, regime features) 기반 LSTM 또는 TCN/Transformer-lite 실험
- **DoD**: Rule-only 대비 OOS IC 또는 hit-rate 개선(통계 유의수준 포함)
- **난이도**: High

### Step 3-2. 텍스트 신호 품질 개선
- **목적**: 키워드 기반 감성의 한계 보완
- **구현 내용**: 금융 도메인 분류기(뉴스 relevance, event type), LLM 출력 스키마 검증(pydantic), confidence gating
- **DoD**: 파싱 실패율 < 1%, low-confidence 신호 자동 제외
- **난이도**: Mid

### Step 3-3. RAG 리빌드
- **목적**: 검색 품질과 설명가능성 향상
- **구현 내용**: chunking 개선(문단 기반), 재랭킹, source attribution, dual-embedding 통합 정책(bge vs gemini 비교 실험)
- **DoD**: 검색 정밀도 기준치 충족, 동일 질의 결과 안정성 확보
- **난이도**: High

### Step 3-4. 앙상블 의사결정
- **목적**: 단일 모델 실패 리스크 축소
- **구현 내용**: rule score + LSTM probability + macro regime를 메타모델로 결합, 불확실성 기반 trade/no-trade 정책
- **DoD**: 단일모델 대비 MDD 감소 + alpha 유지
- **난이도**: High

### Phase 3 체크포인트
- LSTM/시계열 모델이 "실제 개선"을 보여야 Phase 4 진행
- 텍스트/RAG 파이프라인 신뢰도 지표 확보

---

## Phase 4 — 리스크 관리 고도화 & 운영 체계 완성 (예상 6~8주)

### Step 4-1. 포지션 사이징 엔진
- **목적**: 손실 분포 통제
- **구현 내용**: ATR/변동성 타깃 기반 sizing, 계좌 위험예산(예: trade당 0.5~1% VaR)
- **DoD**: 전략 전체 MDD와 개별 포지션 손실 상한이 일관되게 유지
- **난이도**: High

### Step 4-2. 상관관계/섹터 분산 제어
- **목적**: 동시 손실(집중 리스크) 방지
- **구현 내용**: rolling correlation matrix, 섹터/테마 익스포저 캡, 동일 팩터 과집중 제한
- **DoD**: 상관집중 지표(예: 상위 3포지션 공분산 기여도) 임계치 이하
- **난이도**: Mid

### Step 4-3. 실시간 모니터링 & 알림
- **목적**: 운영 이상 탐지 시간 단축
- **구현 내용**: Prometheus/Grafana, 주문 실패율/지연/슬리피지 대시보드, 알림 룰셋
- **DoD**: 핵심 SLO(주문 성공률, 배치 지연, 데이터 품질) 대시보드 상시 가시화
- **난이도**: Mid

### Step 4-4. 복구/연속성 시나리오
- **목적**: 장애 시 자동 복원
- **구현 내용**: DB 백업/복구 리허설, idempotent job 재실행, emergency stop 스위치, 수동 개입 runbook
- **DoD**: 모의 장애 훈련에서 RTO/RPO 목표 달성
- **난이도**: Mid

### Phase 4 체크포인트
- Risk emergency stop 실제 동작 검증
- 운영팀이 무중단으로 장애 대응 가능

## Phase 5 — Paper → Live 전환 및 규제/컴플라이언스 정착 (예상 8~12주)

### Step 5-1. Paper Trading 승급 게이트 정의
- **목적**: 감이 아닌 데이터로 라이브 전환 결정
- **구현 내용**: 최소 관찰 기간(예: 3~6개월), 성과 기준(월간 alpha>0, MDD<15%, 체결오류율<1%)
- **DoD**: 승급 심사 체크리스트 통과 시에만 소액 라이브 허용
- **난이도**: Mid

### Step 5-2. 단계적 라이브 배치
- **목적**: 초기 라이브 리스크 최소화
- **구현 내용**: 자본 5%→15%→30% 단계 증액, 단계별 손실 한도/자동 롤백
- **DoD**: 각 단계 KPI 충족 시 다음 단계 진입
- **난이도**: Mid

### Step 5-3. 법적/규제 고려(미국/한국)
- **목적**: 규제 리스크 사전 차단
- **구현 내용**:
  - 미국: 브로커 API 약관 준수, 투자자문 해당 여부 검토, 로그/감사추적 보존
  - 한국(해외투자): 해외주식 투자 관련 세무/신고, 외화송금/계좌 규정 준수
  - 개인정보/접근통제 정책 정비
- **DoD**: 법무/세무 체크리스트 서면 완료, 위반 가능 경로 제거
- **난이도**: High

### Step 5-4. 운영 거버넌스
- **목적**: 전략 변경이 통제 없이 배포되는 문제 방지
- **구현 내용**: 전략 변경 승인 프로세스, 실험/배포 분리, 월간 리스크 위원회 리뷰
- **DoD**: 모든 전략 변경이 실험 결과 + 승인 이력과 함께 배포
- **난이도**: Mid

### Phase 5 체크포인트 (최종)
- 3개월 연속 월간 alpha 양수
- 롤링 12개월 추정 MDD 15% 이내
- 장애/보안/컴플라이언스 운영지표 안정화

## 최종 현실 진단 요약

현재 시스템은 아이디어와 구현 에너지는 충분하지만, 자동매매 시스템에서 가장 중요한 세 가지(검증, 리스크 통제, 운영탄력성)가 약합니다. 따라서 지금 당장 라이브 대규모 자금 운용은 비합리적입니다. 다만 로드맵대로 **Phase 1~3에서 검증 인프라와 신호 신뢰도**, **Phase 4에서 리스크 엔진과 운영 체계**, **Phase 5에서 승급 게이트와 규제 정합성**을 확보하면 "실전 운용 가능한 자동 트레이딩 플랫폼"으로 진입할 수 있습니다.

핵심은 모델을 더 붙이는 것이 아니라, **검증 가능성과 손실 통제 가능성**을 먼저 만드는 것입니다.
