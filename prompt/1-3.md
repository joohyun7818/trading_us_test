
당신은 Python/알고트레이딩 전문 개발자입니다.

## 배경

`api/services/trading_engine.py`의 `check_exit()` 함수는 `portfolio` 테이블의 `unrealized_pnl_pct`를 참조하여 stop-loss(-8%)를 판단합니다. 그런데 이 테이블은 장 마감 시(15:30) `close_batch`에서만 갱신되므로, 장중 급락 시 stop-loss가 작동하지 않습니다.

`auto_trader.py`의 `auto_trade_loop()`은 5분마다 실행되지만, exit 체크도 portfolio 테이블의 stale 데이터에 의존합니다.

## 작업 요구사항

1. `auto_trader.py`의 exit 체크 로직을 수정하세요:
   * `portfolio` 테이블 대신 `alpaca_client.get_positions()`의 실시간 데이터를 사용합니다.
   * Alpaca 포지션의 `unrealized_plpc` 필드를 직접 읽어 stop-loss/take-profit을 판단합니다.
2. `trading_engine.py`의 `check_exit()` 함수를 리팩토링하세요:
   * 새 파라미터 `current_positions: list[dict]`를 받아 Alpaca 실시간 포지션을 사용합니다.
   * stop-loss, take-profit 임계값을 settings 테이블에서 로드하도록 변경하세요 (하드코딩 -8%, +15% 제거).
3. `auto_trade_loop()` 시작 시 Alpaca 포지션을 한 번 조회하고, 각 포지션에 대해 개선된 `check_exit()`을 호출하세요.
4. stop-loss 발동 시 로깅과 함께 trades 테이블에 `trigger_reason` 컬럼 값을 기록하세요.

## 제약

* `get_positions()`는 동기 SDK 호출이므로 `asyncio.to_thread()`로 감싸세요.
* Alpaca API 실패 시 stale portfolio 데이터로 폴백하되, 로그에 경고를 남기세요.
* 기존 `check_exit()` 시그니처를 유지하면서 새 오버로드를 추가하는 방식도 허용합니다.

## 출력

1. 수정된 `api/services/trading_engine.py` 전체
2. 수정된 `api/services/auto_trader.py` 전체
