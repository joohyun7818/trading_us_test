
AlphaFlow US 프로젝트에 동적 청산(exit) 시스템을 구현해라.

■ 배경 (필독)
현재 backtester.py의 청산 로직은 고정 stop_loss_pct(-8%)와 take_profit_pct(+15%)만 사용한다.
백테스트 결과 모든 거래가 이 두 가격에서만 종료되며, trading_engine.py의 check_exit()에 있는
numeric_score 기반 청산은 백테스트에서 한 번도 발동하지 않았다.
일부 포지션이 11개월 보유되었고, 이는 스윙 트레이딩이 아니라 SL/TP 룰렛이다.

■ 할 일

1) api/services/exit_manager.py 신규 생성

   - async def evaluate_exit(position: dict) -> dict 함수 작성
   - position 인자: symbol, entry_price, current_price, highest_price_since_entry,
     days_held, entry_atr, quantity
   - 청산 규칙 (우선순위 순, 먼저 걸리면 즉시 반환):
     a) ATR 하드 스탑: current_price <= entry_price - (entry_atr × hard_stop_mult)
     → 전량 청산, exit_reason="atr_hard_stop"
     b) 트레일링 스탑: current_price <= highest_price_since_entry - (entry_atr × trail_mult)
     단, highest_price_since_entry > entry_price × 1.01 일 때만 활성
     → 전량 청산, exit_reason="trailing_stop"
     c) 시간 기반: days_held >= max_holding_days
     → 전량 청산, exit_reason="time_limit"
     d) 부분 익절: (current_price - entry_price) / entry_atr >= partial_mult 이고 quantity > 1
     → 50% 청산, exit_reason="partial_take_profit"
     e) 해당 없음 → should_exit=False, exit_reason="hold"
   - hard_stop_mult, trail_mult, max_holding_days, partial_mult는 settings 테이블에서 읽되,
     기본값은 각각 2.5, 2.0, 20, 3.0
   - 반환 형태: {"should_exit": bool, "exit_reason": str, "exit_quantity": int, "details": str}
2) api/services/backtester.py 수정

   - BacktestConfig에 필드 추가:
     exit_strategy: str = "fixed"  (기존 호환)
     hard_stop_atr_mult: float = 2.5
     trailing_stop_atr_mult: float = 2.0
     max_holding_days: int = 20
     partial_exit_atr_mult: float = 3.0
   - run_backtest 내 포지션 관리 루프 수정:
     · positions dict의 value에 "highest_price", "entry_atr" 키 추가
     · 매일 highest_price = max(highest_price, 당일 high)
     · entry_atr = 진입일의 atr_14 값 (없으면 entry_price × 0.02)
     · exit_strategy == "dynamic"이면:
     기존 고정 SL/TP 체크 대신 evaluate_exit 로직을 동기 버전으로 인라인 실행
     (exit_manager는 async지만 backtester는 DB 조회 없이 파라미터만 쓰므로
     동일 로직을 순수 함수로 복제하거나, 파라미터를 직접 전달)
     · exit_strategy == "fixed"이면 기존 로직 유지 (하위 호환)
   - trades 딕셔너리에 "exit_reason" 키 추가
3) api/services/auto_trader.py 수정

   - auto_trade_loop의 exit 체크 부분에서:
     기존 check_exit 호출을 exit_manager.evaluate_exit로 교체
     · Alpaca 포지션에서 highest_price를 추적하려면 portfolio 테이블에 highest_price 컬럼 필요
     · 없으면 current_price를 highest로 간주 (보수적)
     · entry_atr는 trades 테이블 또는 portfolio에서 진입 시 저장
4) DB 마이그레이션 (scripts/ 또는 마이그레이션 SQL)

   - trades 테이블: ALTER TABLE trades ADD COLUMN IFNOT EXISTS exit_reason VARCHAR(30);
   - trades 테이블: ALTER TABLE trades ADD COLUMN IFNOT EXISTS entry_atr FLOAT;
   - portfolio 테이블: ALTER TABLE portfolio ADD COLUMN IFNOT EXISTS highest_price FLOAT;
   - portfolio 테이블: ALTER TABLE portfolio ADD COLUMN IFNOT EXISTS entry_atr FLOAT;
   - settings 초기값 INSERT:
     hard_stop_atr_mult=2.5, trailing_stop_atr_mult=2.0,
     max_holding_days=20, partial_exit_atr_mult=3.0

■ 제약

- 프로젝트 코딩 규칙 준수: async def, asyncpg, type hints, 한국어 docstring, logging
- exit_strategy="fixed"일 때 기존 백테스트 결과와 100% 동일해야 함 (회귀 안전)
- 새 파일은 api/services/ 아래에 생성

■ 완료 기준

- exit_strategy="dynamic"으로 백테스트 실행 시 exit_reason에 atr_hard_stop, trailing_stop,
  time_limit, partial_take_profit 네 종류가 최소 1건씩 나타남
- exit_strategy="fixed"로 실행 시 기존 결과와 동일
- 평균 보유 기간이 20일 이하
