## 1) 아키텍처 설계 — **등급: B-**

### 핵심 문제
- **스케줄 의존성 충돌 가능성**: `api/services/scheduler.py`에서 10개 잡이 독립 등록되지만, 데이터 의존성(예: 뉴스 수집 → 인덱싱 → RAG 분석)이 스케줄 단위로 강하게 보장되지 않습니다. 같은 시간대에 `news_crawl`, `rag_index`, `gemini_index`, `auto_trade`가 겹칠 수 있어 결과 일관성이 흔들릴 여지가 큽니다.
- **에러 격리 약함**: 잡별 `try/except`로 로깅은 하지만 실패 전파/알림/자동 복구 루프가 없습니다. `main.py` lifespan에서도 DB/Chroma/scheduler 초기화 실패를 로그만 남기고 계속 기동할 수 있어 “부분 장애 상태로 정상 서비스처럼 보이는” 위험이 있습니다.
- **중복/충돌 코드 존재**: `api/services/macro_engine.py`에 `calculate_regime()`가 중복 정의되어 뒤쪽 정의가 앞쪽 정의를 덮어씁니다. 앞쪽에는 geopolitical 반영 버전이 있으나 실제 런타임에는 뒤쪽 버전이 사용될 가능성이 높습니다. 이건 설계/운영 둘 다에 치명적인 기술부채입니다.
- **동시성 제어의 한계**: `auto_trader.py`의 `_trading_running`/`_leveraged_running` 플래그는 단일 프로세스 기준 보호만 가능하며, 멀티프로세스/재기동/중복 실행 시 안전하지 않습니다.

### 상대적 강점(제한적)
- FastAPI + asyncpg + APScheduler 조합 자체는 합리적이며 모듈 분리는 대체로 명확합니다.
- `ollama_client.py`의 `asyncio.Lock`으로 모델 호출 직렬화는 로컬 LLM OOM/리소스 경합을 줄이는 실무적 선택입니다.

## 2) 데이터 파이프라인 품질 — **등급: C+**

### 핵심 문제
- **가격 데이터 근본 리스크**: `price_crawler.py`는 yfinance 기반 일봉/과거 데이터 수집 중심입니다. 실거래 판단에 필요한 저지연 체계가 아니며, API 지연/누락 시 stale 데이터에 의존할 수 있습니다.
- **중복 뉴스 방지 한계**: `news_crawler.py`는 `url_hash` 기반 중복 제거만 수행합니다. 같은 기사라도 URL이 다르면 중복으로 들어올 수 있어 감성 점수가 사건 수보다 커질 가능성이 큽니다.
- **결측 처리의 방어성 부족**: `price_crawler.py`에서 결측 시 스킵이 많아 일부 종목은 지표 갱신 누락 상태로 남을 수 있습니다. 누락 종목/비율에 대한 품질 지표가 없습니다.
- **데이터 파이프라인 품질지표 부재**: 수집 성공률, 소스별 오류율, 임베딩 누락률, 지연(latency) 분포를 관리하는 계량 지표가 거의 없습니다.

### 상대적 강점(제한적)
- 뉴스 소스 다변화(Finnhub + Yahoo RSS + Google RSS)는 단일 소스 리스크를 낮춥니다.
- DB 스키마에서 `url_hash UNIQUE`, `embedded` 플래그 등 기본적인 파이프라인 제어 지점은 갖췄습니다.

## 3) AI/ML 모델 활용 — **등급: C**

### 핵심 문제
- **1차 감성의 구조적 한계**: `sentiment.py`는 키워드 카운팅 기반이라 문맥/부정문/풍자/도메인 문법에 취약합니다. “실전 유효성”을 입증할 검증 루프가 없습니다.
- **RAG 출력 신뢰성 낮음**: `rag_analyzer.py`의 파싱은 JSON 파싱 실패 시 정규식 추출로 내려가는데, 이 fallback은 쉽게 왜곡된 필드를 통과시킬 수 있습니다.
- **입력/출력 스키마 강제 약함**: 모델 출력의 범위 검증(예: sentiment_score 범위) 및 strict schema enforcement가 충분하지 않습니다.
- **비전 분석의 운영 적합성 낮음**: `chart_analyzer.py`는 4시간 캐시(`analysis_cache`)를 고정 사용합니다. 스윙이라도 이벤트 급변 구간에서는 stale 시각 신호가 누적됩니다.
- **이중 임베딩 컬렉션 운영 혼선**: `news_indexer.py`(bge-m3/ollama embed) + `gemini_indexer.py`(gemini-embedding-001) 병행 구조는 실험 단계로는 유효하지만, 프로덕션에서 “어떤 컬렉션이 의사결정 기준인지”가 명확하지 않습니다.

### 상대적 강점(제한적)
- 로컬 LLM + 임베딩 + RAG + 비전까지 통합 파이프라인을 만든 점은 PoC 확장성 측면에서 의미가 있습니다.
- LLM 실패 시 기본값 반환 등 서비스 연속성 방어 코드는 존재합니다.

## 4) 트레이딩 로직(핵심) — **등급: C**

### 핵심 문제
- **임계값 근거 부족**: Step1의 `score >=60 or <=35`(batch.py), 최종 `BUY>=70/SELL<=30`(trading_engine.py) 기준은 존재하지만 데이터 기반 최적화/통계 검증 근거가 코드에 없습니다.
- **가중치 타당성 검증 부재**: `text 35 / numeric 50 / macro 15`는 설정값으로 잘 분리되어 있으나, 리밸런싱 근거(시장 국면별 성능 비교)가 없습니다.
- **실행 순서 편향 가능성**: `auto_trader.py`에서 `ORDER BY final_score DESC`로 미실행 신호를 처리합니다. 동일 점수 다수 발생 시 우선순위가 불명확하고 체결 예산 소진 순서 편향이 생길 수 있습니다.
- **리스크 관리 단순화**: stop-loss -8%, take-profit +15%는 고정형이며 종목 변동성(ATR), 섹터 상관, 포트폴리오 군집 리스크를 반영하지 않습니다.
- **레버리지 ETF 전략 현실성 문제**: TQQQ/SQQQ 전략은 변동성 끌림(volatility drag), 갭 리스크, 급변장 체결 슬리피지에 민감한데 현재 제어로직은 비교적 단순합니다.

### 상대적 강점(제한적)
- `alpaca_client.py`에 `max_order_amount`, `daily_order_limit`, `max_exposure_pct` 같은 1차 안전장치는 구현되어 있습니다.

## 2) 성숙도 단계 판정
**판정: MVP (Production 아님)**

근거:
1. 실제 주문 실행 파이프라인은 구현되어 있음(=PoC는 넘음).
2. 그러나 전략 검증(백테스트/워크포워드), 품질 게이트, 장애 대응, 보안·운영이 프로덕션 수준에 못 미침.
3. 특히 중복 정의/테스트 부재/운영 알림 부재는 Beta 상용 운영의 최소 요건도 충족하지 못합니다.

## 3) “실제 돈을 걸어도 되는가?”
**결론: No**

구체적 근거:
- **정량 근거 부재**: 현재 임계값·가중치가 통계적으로 최적화되었다는 증거가 없습니다.
- **회귀 안전성 부재**: 테스트가 없어 변경 시 전략 왜곡 여부를 사전에 잡기 어렵습니다.
- **운영 리스크 높음**: 장애 감지/알림/복구 경로가 약해, 오작동이 길게 지속될 수 있습니다.
- **보안 리스크 존재**: 운영 CORS/비밀 관리/기본 자격 정보 등 기본 보안 태세가 약합니다.
- **레버리지 리스크**: TQQQ/SQQQ는 고난도 상품인데 현재 리스크 모델은 고정 임계값 중심입니다.

## 4) 가장 시급한 Top 5 문제 (우선순위)
1. **백테스트+워크포워드 부재 (P0)**  
   - 전략의 생존 가능성을 입증할 수 없음. 지금은 “좋아 보이는 규칙 집합” 상태.
2. **macro_engine 중복 정의 제거 및 레짐 계산 단일화 (P0)**  
   - 의도와 실제 실행이 어긋날 수 있는 구조적 버그.
3. **자동매매 동시성/멱등성 강화 (P1)**  
   - DB 락/idempotency key/중복 실행 차단 없으면 중복 주문 위험.
4. **운영 안전장치 구축 (P1)**  
   - 장애 알림, 헬스 체크 고도화, 비상중단, 실패 시 거래 중단 정책 필요.
5. **테스트 체계 도입 (P1)**  
   - 최소 단위테스트(스코어링·리스크·시그널 결정) + 통합테스트(주문 시뮬레이션) 없이는 배포 불가.

# Part 3: 실전 로드맵 (Live Trading 가능 수준까지)

최종 목표: **월간 양의 알파를 생성하면서 최대 낙폭(MDD) 15% 이내 통제**

아래는 최소 5단계 로드맵입니다. 기간은 병렬 작업 일부를 고려한 현실적 추정입니다.

---

## Phase 1 (4~6주): 전략 검증 기반 구축 (Backtest First)

### Step 1-1. 이벤트 정렬형 백테스트 엔진 구축
- 목적: 현재 규칙(스크리닝/가중치/임계값)이 과거 데이터에서 유효한지 검증
- 구현 내용:
  - 시점 정렬(뉴스 publish time, 지표 산출 가능 시점, 거래 가능 시점) 엄격화
  - 거래비용/슬리피지/부분체결 가정 반영
  - BUY/SELL/HOLD 및 레버리지 로직 재현 가능 엔진 작성
- DoD:
  - 동일 입력 데이터로 결과 재현성 확보(결과 편차 0)
  - 3년 이상 구간, 연도별 성능 리포트 자동 생성
- 난이도: **High**

### Step 1-2. 파라미터 탐색(가중치/임계값) + 과최적화 방지
- 목적: 35/50/15, 70/30, 60/35의 데이터 기반 재검증
- 구현 내용:
  - Grid/Bayesian 탐색 + walk-forward 검증
  - 성능 지표: CAGR, Sharpe, Calmar, MDD, turnover, hit ratio
- DoD:
  - 학습/검증/테스트 구간 분리 결과 제출
  - OOS(out-of-sample)에서 Sharpe>0.8, MDD<20% 1차 달성
- 난이도: **High**

### Step 1-3. 성능 기준선(Benchmark) 고정
- 목적: 전략 개선 효과를 객관화
- 구현 내용:
  - SPY buy&hold, equal-weight, sector-neutral baseline 동시 비교
- DoD:
  - 모든 실험 결과에 baseline 대비 alpha/IR 자동 표기
- 난이도: **Mid**

**Phase 1 의존성/체크포인트**
- 의존성: 정제된 히스토리컬 데이터, 거래비용 가정 정의
- 체크포인트: “백테스트 재현성 + OOS 성능 기준” 미달 시 이후 Phase 진행 금지

## Phase 2 (3~5주): 시그널 품질 검증 파이프라인 + 품질 게이트

### Step 2-1. Signal QA 파이프라인 구축
- 목적: 라이브 신호 품질을 지속적으로 계량
- 구현 내용:
  - 신호별 1D/5D/10D 성과, 섹터별 성과, 점수-수익률 상관 추적
  - 신호 생성 시점과 체결 시점 분리 기록
- DoD:
  - 일/주 자동 리포트 + 성능 드리프트 감지 알람
- 난이도: **Mid**

### Step 2-2. LLM 출력 검증 강제
- 목적: LLM 파싱 오류/환각이 주문으로 전파되는 것 차단
- 구현 내용:
  - Pydantic strict schema, 범위 검증, invalid output fallback 정책
  - 프롬프트/응답 감사 로그 구조화
- DoD:
  - invalid output 주문 전파 0건
  - 파싱 실패율 대시보드화
- 난이도: **Mid**

### Step 2-3. 데이터 신선도/정합성 모니터링
- 목적: stale 데이터 기반 의사결정 방지
- 구현 내용:
  - 가격/뉴스/임베딩별 freshness SLA 정의
  - 임계 초과 시 자동 HOLD 또는 거래 중단
- DoD:
  - freshness 위반 시 강제 safe mode 동작 검증
- 난이도: **Mid**

**Phase 2 의존성/체크포인트**
- 의존성: Phase 1 성능 기준선/검증 리포트
- 체크포인트: 최근 4주 신호 품질 하락(예: hit ratio 급락) 시 전략 재학습 루프로 회귀

---

## Phase 3 (5~8주): LSTM/시계열 모델 도입 + 앙상블

### Step 3-1. 시계열 예측 모델 프로토타입(LSTM/Temporal Transformer)
- 목적: 룰 기반 점수의 정보 손실 보완
- 구현 내용:
  - 입력: OHLCV + 기술지표 + 뉴스 임베딩 요약 + 매크로/지정학 피처
  - 타깃: n일 수익률 분포 또는 상승확률
- DoD:
  - OOS 기준 룰베이스 대비 예측력 개선(AUC/IC/Hit ratio)
- 난이도: **High**

### Step 3-2. 룰+모델 앙상블 정책 수립
- 목적: 단일 모델 리스크 완화
- 구현 내용:
  - 룰 점수와 모델 확률을 결합한 메타 스코어
  - 불확실성(분산) 기반 포지션 축소 규칙
- DoD:
  - 백테스트 OOS에서 Sharpe 개선 + MDD 감소 동시 달성
- 난이도: **High**

### Step 3-3. 모델 운영(MLOps) 최소 체계
- 목적: 모델 드리프트/재학습 자동화
- 구현 내용:
  - 데이터 버전 관리, 주기 재학습, 모델 레지스트리, 배포 승인 게이트
- DoD:
  - 재학습 파이프라인 자동 실행 + 롤백 가능
- 난이도: **High**

**Phase 3 의존성/체크포인트**
- 의존성: Phase 1~2에서 정합한 학습 데이터셋과 품질지표 확보
- 체크포인트: 모델 성능이 룰 대비 열위면 도입 보류(강행 금지)

---

## Phase 4 (4~6주): 리스크 관리 고도화 (목표 MDD 15%)

### Step 4-1. 포지션 사이징 엔진
- 목적: 종목별/시장상황별 위험기여 통제
- 구현 내용:
  - ATR 변동성 기반 사이징, Kelly fraction capped, 종목당/섹터당 상한
- DoD:
  - 단일 종목 손실이 계좌 전체에 미치는 영향 상한 수치화
- 난이도: **High**

### Step 4-2. 상관관계/섹터 분산 제약
- 목적: 같은 베팅의 중복 노출 방지
- 구현 내용:
  - rolling correlation, PCA risk cluster 기반 동시 포지션 제한
  - 섹터 편중 상한(예: 25%)
- DoD:
  - 스트레스 시나리오에서 포트폴리오 VaR/ES 개선 확인
- 난이도: **High**

### Step 4-3. 동적 손절/익절 + 비상중단
- 목적: 고정 -8/+15 한계를 극복하고 급락 방어
- 구현 내용:
  - 변동성/레짐 기반 동적 손절
  - 일손실 한도, 연속 손실 횟수, 시스템 이상 시 자동 거래중지
- DoD:
  - 시뮬레이션에서 MDD 15% 이내 달성
- 난이도: **Mid~High**

### Step 4-4. 레버리지 ETF 별도 리스크 프레임
- 목적: TQQQ/SQQQ 특수 리스크 격리
- 구현 내용:
  - 별도 예산계정(capital bucket), 보유기간/갭리스크 제한 강화
  - 레짐 신뢰도 낮을 때 자동 비활성화
- DoD:
  - 레버리지 전략 단독/통합 성과 보고서 분리
- 난이도: **Mid**

**Phase 4 의존성/체크포인트**
- 의존성: 안정적인 신호 품질(Phase 2) + 모델/룰 결합 결과(Phase 3)
- 체크포인트: 실거래 전 모의계좌에서 8~12주 연속 MDD<15%

## Phase 5 (3~4주 + 최소 12주 관찰): 운영 고도화 + Paper→Live 전환

### Step 5-1. 모니터링/알림 체계 구축
- 목적: 장애 탐지·대응 속도 확보
- 구현 내용:
  - 메트릭: 수집 지연, 시그널 수, 주문 성공률, PnL, 레짐 변화율
  - 알림: Slack/PagerDuty/이메일, 심각도별 Runbook 연결
- DoD:
  - 장애 모의훈련에서 탐지→조치 시간(SLA) 충족
- 난이도: **Mid**

### Step 5-2. 보안/감사/복구 체계
- 목적: 운영/규제 리스크 최소화
- 구현 내용:
  - 비밀관리(Secret Manager), CORS 축소, 인증/권한, 감사로그
  - DB 백업/복구 리허설, 배포 롤백 절차
- DoD:
  - 보안 점검 체크리스트 통과, 복구 리허설 성공
- 난이도: **Mid**

### Step 5-3. Paper Trading 게이트 운영 (최소 12주)
- 목적: Live 전환 전 실전 조건 검증
- 구현 내용:
  - 사전 정의된 게이트:
    - Sharpe >= 1.0 (거래비용 반영)
    - 월간 alpha > 0
    - MDD <= 15%
    - 주문 실패율/시스템 장애율 임계 이하
- DoD:
  - 12주 연속 게이트 충족 시에만 Live 소액 전환 승인
- 난이도: **Mid**

### Step 5-4. Live 전환 단계적 확대
- 목적: 초기 손실과 운영 사고 리스크 제한
- 구현 내용:
  - 1단계 1% 자본 → 2단계 3% → 3단계 5% (각 단계 최소 4주 관찰)
  - 단계별 실패 시 즉시 이전 단계로 롤백
- DoD:
  - 각 단계 KPI 충족 + 운영사고 0건
- 난이도: **Mid**

**Phase 5 의존성/체크포인트**
- 의존성: Phase 1~4 완료 + 모의계좌 장기 성과 검증
- 체크포인트: 어떤 게이트라도 미충족이면 Live 확대 중단

## 법적/규제 고려사항 (미국 + 한국)

> 아래는 법률 자문이 아닌 실무 체크리스트입니다. 실제 적용 전 미국/한국 전문가 자문 필수.

1. **미국 증권 규제**
- 본인계좌 자동매매와 타인자금 운용은 규제 강도가 다릅니다.
- 타인 자금/자문 성격이 개입되면 SEC/FINRA/주(州) 규제 이슈가 발생할 수 있습니다.
- 주문/의사결정/리스크 룰의 감사가능 로그 보존이 중요합니다.

2. **브로커/약관 준수**
- Alpaca API 이용약관, 주문빈도 제한, 데이터 라이선스 준수 필요.
- 장애 시 자동주문 반복/과도호출은 계정 제한 리스크가 있습니다.

3. **한국 거주자의 해외투자 규정**
- 해외주식 투자 관련 세무 신고(양도소득, 배당) 및 외환 관련 규정 검토 필요.
- 법인/개인 구조에 따라 회계/세무 처리 방식이 달라집니다.

4. **내부통제 권고**
- 전략 변경 승인 프로세스(변경요청-검토-배포-검증)
- 손실 한도/운영 중단 권한 명확화
- 정기 리스크 리뷰(주간/월간) 문서화

현재 AlphaFlow US는 “아이디어가 많은 자동매매 MVP”입니다. 하지만 라이브 자금을 맡길 수준의 검증/운영 통제가 부족합니다. 먼저 백테스트·검증·리스크·운영 게이트를 완성하고, **Paper 성과로 증명한 뒤 단계적으로 Live 전환**해야 합니다. 목표(월간 양의 알파 + MDD 15% 이내)는 가능하지만, 지금 코드 상태에서는 바로 달성한다고 가정하면 위험합니다.
